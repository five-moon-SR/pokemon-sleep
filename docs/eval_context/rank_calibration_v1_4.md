## v1.4 の設計（%は不変、ランク帯だけ差し替え）

### 母集団
`scripts/calibrate_rank_thresholds.py`:
- 全種 × 品質4段のLv60合成個体（計964体）
- 品質ビルドは**種族の得意分野に噛み合わせる**（_AXIS_BUILD: きのみ=きのみの数S系/食材=食材確率M系/スキル=スキル確率M系）。
  ※噛み合わせないと食材型の帯が2ptに圧縮される（初回の失敗。ミスマッチビルドは上限が低い）
- 品質: plain(無補正,skl3) / mid(白サブ1,skl4) / good(適正金1+スピM,skl6) / great(適正金2+M系,skl6)

### 品質アンカー（閾値の意味論）
```
増田 = great の上位10%   （理想ビルドのさらに上澄み）
SS   = great の中央値     （理想ビルド級）
S    = (good中央 + great中央) / 2  ＝「明確に良く仕上がった個体」
A    = good の中央値      （金1+スピM級）
B    = mid の中央値       （並・白サブ級）
C    = plain の中央値     （無補正級）
D    = それ未満
```
- 0.5刻み丸め、最低2ptの帯間隔を強制
- 偶数タイプ(2/4/6/8)は個体のdaifuku_eval_type登録時のみ出現し標本ゼロ→**全体分布で代用**
- タイプ7はn=10種と少ない（代用はしていないが要注意）

### 採用値
constants.py `RANK_THRESHOLDS_BY_TYPE`（v1.4値）。旧だいふく公式帯は
`DAIFUKU_RANK_THRESHOLDS_BY_TYPE` として参照保持。`EVALUATION_VERSION = "v1.4"`。

### 新帯での分布（汎用ビルド個体で再検証）
- 無補正: C/D ／ 汎用金1: ほぼB（A:33）／ 汎用スキル寄せ優個体: S:29 増田:10（スキル型のみ）
- 検証個体リザードン60.0% → **B**（だいふくならS。きのみ寄りサブが食材型⑤に不適合だから、が新解釈）

## 再編集時の論点・ツマミ

1. **厳しすぎ/緩すぎの調整**: `calibrate_rank_thresholds.py` の anchors（percentileとS中間点式）を変更→再実行→出力を constants に貼替。例: Sを緩めるなら `S = good_p75` に
2. **偶数タイプの代用問題**: daifuku_eval_type持ち個体が偶数タイプだと全体分布代用の帯になる。実データで違和感が出たら、偶数タイプは「隣接奇数タイプの帯」を使う案あり
3. **option_bonus は未調整**: -10〜+30がそのまま加算されている（金種サブ加点・スキルLvUP節約・EXP性格）。ランク帯より先にここが過剰の可能性もある（great→増田の押し上げ役）。上限を+15に絞る案
4. **現状評価がLv基準で辛い**: ベンチマークがLv60固定なので低Lv個体は10-20%台=D。これは仕様（伸びしろはLv50/60ランクで見る）だが、「現状ランク」の表示意義は再考の余地
5. **軸ミスマッチの解釈**: 新帯では「種族タイプに噛み合わないサブ構成」はBに沈む。ミスマッチでも汎用的に強いビルド（スピM等）を救うなら、タイプ別でなく「その個体の最良軸」で評価する案
6. **だいふく互換の担保**: %表示はだいふくと一致しているので触らないのが吉。ズレたらまずPlaywrightで実サイト照合（このファイルの手順で再現可）
7. **skill_effects の換算係数**（きのみ1個=100/食材=144.8/おてつだい1回=200/大成功1%=200/げんき1%=80·200）は粗いヒューリスティック。スキル軸の評価を精密化するならここから
